# Phase 2: 简历库系统实现

## 背景调研结论
- 输入类型：`detailed_plan`
- 当前现状：Phase 1 已完成 PDF 上传、MinerU 解析、LLM 分析、会话管理的完整链路。Bot 通过飞书 WebSocket 实时接收消息，Handler 框架已就绪。
- 已确认计划背景：需要构建简历索引与检索系统。支持人力专员通过自然语言对话搜索候选人。双通道设计（ChromaDB 向量 + SQLite FTS5 关键词）已确认，但 Embedding 模型和 Reranker 模型尚未部署，因此本阶段先实现 FTS5 关键词检索全链路。
- 约束核对：
  - Embedding 模型未部署 → ChromaDB 向量搜索暂缓，先搭好框架（占位）
  - Reranker 未部署 → 暂不涉及
  - Docker 容器内运行，无需外部网络
  - 千份简历级别，无需考虑分布式
- 建议修改方向：分三阶段实现：
  1. SQLite + FTS5 基础设施 + jieba 分词
  2. 入库索引链路（resume_handler → 分词 → FTS5 写入）
  3. 检索链路（LLM 关键词提取 → FTS5 MATCH → 结果注入 LLM）
- Phase 划分依据：按「数据层 → 写入流 → 读取流」三层递进，每层可独立测试和验证。

## Phase 1：SQLite + FTS5 基础设施搭建

### 任务背景
当前系统使用 JSON 文件存储会话，没有结构化的简历数据库。需要建立 SQLite 数据库，包含：
- `resumes` 主表：存储简历元数据、文件路径、去重
- `resumes_fts` 虚拟表：FTS5 全文索引
- jieba 分词初始化

### 预期达到的任务效果
- `services/db.py` 提供数据库连接管理（单例连接）
- 启动时自动创建表结构（含 FTS5 虚拟表）
- jieba 分词器可用，`cut_for_search` 可正常调用并返回分词结果
- 可通过 Python 代码对简历表和 FTS5 表进行增删查

### 预计修改的文件范围
- 新增：`services/db.py`
  - `get_connection()` — 获取 SQLite 连接（单例）
  - `init_db()` — 创建 resumes 表 + resumes_fts 虚拟表 + 索引
  - `jieba_init()` — jieba 加载，确认分词可用
- 修改：`config.py`
  - 新增 `sqlite_path` 配置项（默认 `/app/sessions/resumes.db`）
- 修改：`main.py`
  - 启动时调用 `init_db()` 确保表存在

### 必须涵盖的测试方向
- `uv run python3 -c "from services.db import get_connection, init_db; init_db()"` — 验证建表成功
- `uv run python3 -c "import jieba; print(list(jieba.cut_for_search('复旦大学灵均投资')))"` — 验证分词输出
- 验证 FTS5 表可写入并 BM25 可排序：
  ```python
  conn = get_connection()
  conn.execute("INSERT INTO resumes_fts VALUES (?, ?, ?, ?, ?, ?)", ...)
  conn.execute("SELECT rowid, rank FROM resumes_fts WHERE resumes_fts MATCH ?", ['复旦'])
  ```

### 最终验收标准
- 启动 Bot 后 SQLite 数据库文件生成在预期路径
- `resumes` 表字段齐全（name, sex, phone, email, metadata, chroma_id, pdf_path, markdown_path, created_at）
- `resumes_fts` 表字段齐全（full_text, name, school, skills, company）
- jieba `cut_for_search("复旦大学")` 返回包含 "复旦"、"大学"、"复旦大学"
- FTS5 MATCH 查询正常返回 BM25 排序结果

## Phase 2：简历入库索引链路

### 任务背景
当前 `resume_handler.py` 已完成 MinerU 解析和 LLM 结构化提取，但结果只存入了会话 JSON 文件。需要新增索引步骤，将解析结果写入 SQLite + FTS5，并同步保存文件路径。

### 预期达到的任务效果
- 用户发送 PDF 简历后，经过现有链路（下载 → MinerU → LLM 分析），自动触发入库
- 入库内容：jieba 分词后的全文存入 FTS5，元数据存入 resumes 表
- 支持 UPSERT 去重（name + sex + phone 相同则更新）
- PDF 路径和 Markdown 路径记录在数据库中

### 预计修改的文件范围
- 新增：`services/resume_indexer.py`
  - `index_resume(name, sex, phone, email, ...)` — 主入库函数
    - jieba `cut_for_search` 分词（全文 + 各字段）
    - SQLite UPSERT
    - FTS5 写入（delete old + insert new）
    - ChromaDB 占位（留待 embedding 模型部署后接入）
    - 记录 pdf_path 和 markdown_path
- 修改：`services/handlers/resume_handler.py`
  - LLM 分析完成后，调用 `index_resume()` 进行入库
  - 提取 `name`, `sex`, `phone`, `email`, `undergraduate`, `master`, `doctor`, `skills`, `intership_comps`, `work_comps`
  - 传入 pdf_path 和 markdown_path
- 可能联动：`config.py`
  - 新增 `mineru_process_dir`（已有），确认路径可从 config 读取

### 必须涵盖的测试方向
- `uv run python3 -c "from services.resume_indexer import index_resume; index_resume(...)"` — 验证入库
- 验证 UPSERT：相同 name+sex+phone 重复调用 → 更新而非新增
- 验证 FTS5：入库后 `MATCH '复旦'` 能命中对应记录
- 验证文件路径被正确记录：`SELECT pdf_path, markdown_path FROM resumes WHERE name=?`
- 验证 LLM 提取的字段正确映射到 metadata 字段

### 最终验收标准
- 发 PDF → Bot 解析 → 入库完成后，SQLite 中可查到该简历记录
- FTS5 分词命中：搜索 "复旦"、"灵均"、"CTA" 等都能 BM25 排序返回
- 相同人投递两次 → 只保留一条记录（已更新）
- 路径字段非空，可按路径找到实际文件

## Phase 3：FTS5 检索链路 + LLM 关键词提取

### 任务背景
用户通过自然语言对话查询简历时，当前 TextHandler 只走 LLM 对话流程。需要新增检索分支：解析用户意图 → 提取关键词 → FTS5 搜索 → 结果注入 LLM 上下文。

注意：本阶段仅实现 FTS5 关键词检索。ChromaDB 向量搜索和 Reranker 等待 Embedding 模型部署后再补充。

### 预期达到的任务效果
- 用户发文字消息时，TextHandler 自动尝试检索简历
- 检索子线程：LLM 提取关键词 → 拼 FTS5 MATCH 语句
- FTS5 搜索 → BM25 排序 → top-10
- 检索结果注入 LLM 上下文 → LLM 自行判断是否回答简历问题
- 无匹配结果时正常聊天
- ChromaDB 向量搜索占位（空结果），RRF 占位（只返回 FTS5 结果）

### 预计修改的文件范围
- 新增：`services/resume_searcher.py`
  - `search_resumes(keywords)` — FTS5 MATCH 搜索
    - `build_fts5_query(tokens)` — 关键词转 MATCH 语句（OR 支持、NOT 支持、引号保护）
    - FTS5 执行 BM25 搜索，返回 top-10
    - 关联 `resumes` 表取完整 metadata
    - `merge_results(fts_results, vector_results)` — RRF 占位，当前仅返回 FTS5
- 新增：`services/keyword_extractor.py`
  - `KEYWORD_EXTRACT_PROMPT` — LLM 关键词提取系统提示词
  - `extract_keywords(user_query, llm_client, config)` — 独立 LLM 调用
    - 使用独立的 LLM 上下文（不污染主聊天上下文）
    - 解析 LLM 输出为关键词列表
    - 处理多选（`复旦/交大` → `(复旦 OR 交大)`）
    - 处理否定（`社招/NOT` → `NOT 社招`）
- 修改：`services/handlers/text_handler.py`
  - 在 handle() 中新增检索分支：
    - 子线程：`extract_keywords()`（`concurrent.futures.ThreadPoolExecutor`）
    - 主线程：ChromaDB 占位搜索（返回空）
    - 合并：调用 `merge_results()`
    - 结果注入 LLM 上下文
  - 检索结果为空 → 不注入，正常聊天
  - 检索结果非空 → 注入 → LLM 回答
- 修改：`config.py`
  - 新增 `fts_max_results` 配置（默认 10）
  - 新增 `keyword_extract_model` 配置（默认使用主 LLM）

### 必须涵盖的测试方向
- `uv run python3 -c "from services.resume_searcher import build_fts5_query; print(build_fts5_query(['复旦/交大', 'CTA', '社招/NOT']))"` 
  → 输出 `'(复旦 OR 交大) AND CTA NOT (社招)'`
- 验证特殊字符保护：`build_fts5_query(['C++', 'Python'])` → `'"C++" AND "Python"'`
- 测试空关键词：`build_fts5_query([])` → `None`
- 测试子线程：`ThreadPoolExecutor` 提交 `extract_keywords()` 可正常返回
- E2E：发消息"找复旦的CTA实习生" → 检索到已入库的简历 → LLM 回答包含候选人信息
- E2E：发消息"今天天气怎么样" → 检索结果为空 → LLM 正常聊天

### 最终验收标准
- 自然语言查询可触发 FTS5 检索，返回 BM25 排序结果
- 检索结果正确注入 LLM 上下文，LLM 能基于简历内容回答
- 无检索结果时 LLM 正常聊天，不报错
- 关键词提取子线程与主线程互不阻塞（主线程不等 LLM 就可以先返回）
- 关键词支持：
  - 多选：`复旦/交大` → `(复旦 OR 交大)`
  - 排除：`社招/NOT` → `NOT (社招)`
  - 特殊字符：`C++` → `"C++"`
